# 财务年报复杂表格提取系统

## 1. 概述 (Overview)

本方案旨在构建一套工业级的、专门针对财务年报（PDF）中复杂非结构化表格进行数据提取的系统。针对财务数据“零容错”的特性，系统采用 **“计算与业务分离”** 的微服务架构，并在核心算法上遵循 **“规则优先，模型兜底（Rule-First, Model-Fallback）”** 策略。

在 v2.0 架构中，系统全面升级了**数据质量控制链路**，引入了轻量级语义向量匹配、单位/币种智能归一化、全局置信度评分体系以及基于 BBox 的“所见即所得”数据溯源机制，确保系统不仅能“提得出”数据，更能让审计人员“信得过”数据。

## 2. 核心架构设计 (System Architecture)

### 2.1 架构理念与服务拓扑

系统采用 **“生产者-消费者”** 异步消息驱动模式，解耦高并发的业务请求与计算密集的深度学习解析。

- **业务核心服务 (Core Service - CPU集群)**：系统的“大脑”。负责 API 网关、任务调度、基于轻量级 NLP 模型的语义匹配、Pandas 规则提取、单位归一化、置信度计算。
    
- **智能解析服务 (Parser Service - GPU集群)**：系统的“肌肉”。运行 MinerU (Magic-PDF) 执行版面分析与表格结构识别，输出带有严格物理坐标（BBox）的 `content_list.json`。
    

### 2.2 数据存储层

- **PostgreSQL (JSONB)**: 存储文档目录树、提取规则（锚点库、路径指纹库）、任务状态及最终结构化数据。
    
- **MinIO**: 存储原始 PDF、解析产生的 `content_list.json` 、 PDF 切片图片（可选）。
    

## 3. 核心业务流程与功能规格 (Core Workflow & Specs)

### 3.1 第一阶段：高精度解析与预处理 (Parsing & Pre-processing)

- **目标**: 将无结构的 PDF 转化为带坐标的机器可读数据，并解决跨页表格的物理割裂问题。
    
- **功能规格**:
    
    1. **版面解析与 BBox 绑定**: Parser Service 识别所有文本块和表格，生成的 `content_list.json` 采用 MinerU 官方扁平数组格式；每个内容块至少包含 `type`、`page_idx` 与 `bbox`，其中文本块携带 `text`，表格块携带 `table_body`，并保留 MinerU 原始扩展字段。
        
    2. **连续长表格截断与跨页合并 (Pagination Merge)**:
        
        - _识别逻辑_：若连续两页存在表格，且后一页表格顶部带有“续表”字样，或前一页表格底部无闭合线段且紧接下一页表格，或比对相邻两页表格的表头结构（列数、列名），若完全一致，且上下页在上下文逻辑中连贯，也应判定为同一个逻辑长表。。
            
        - _处理动作_：Parser 保留原始 MinerU 产物，不回写逻辑续表标记。Core Service 在构建 Pandas DataFrame 时，基于相邻页表格内容与上下文自动推断连续关系，执行纵向拼接（`pd.concat`），并**自动剔除后续页重复的表头行**，还原为一个逻辑完整的宽长表。
            

### 3.2 第二阶段：智能路由与语义定位 (Routing & Semantic Anchoring)

- **目标**: 通过提取规则（锚点库、路径指纹库）在上百个表格中精准定位目标表，结合表格内部特征与外部物理上下文，完整提取目标表格的结构化数据（html）
    
- **功能规格**:
    
    1. 宏观路由 —— 匹配“路径指纹库” (粗排 / 降维)

    - **执行动作：** 系统扫描整个 PDF 的目录树（或层级标题），利用“路径指纹库”寻找目标章节。
        
    - **核心目的：** **大幅缩减搜索空间**。一份真实的A股年报动辄 300 页、包含几百个表格。如果不做路径过滤，直接全量进行向量匹配，不仅 CPU 计算压力巨大，还会导致极高的误判率（因为不同章节经常会出现表头完全相同的表格，如“母公司利润表”和“合并利润表”）。
        
    - **结果：** 将搜索范围从“整本年报的 500 个表格”缩小到“目标章节内的 5-10 个候选表格”。
        

    2. 微观定位 —— 匹配“锚点库” (精排 / 确认)

    - **执行动作：** 系统进入上一步锁定的候选区间，提取这 5-10 个候选表格的内部特征（表头、列数）和外部特征（上下文、标题），利用“锚点库”结合轻量级模型（如 `BAAI/bge-m3`）进行复合打分。
        
    - **核心目的：** **精准锁定唯一目标**。在同一章节内，表格的业务口径可能存在细微差别。此时通过锚点库中预设的强排他性特征（如必须包含某几个关键列，或上下文必须提及特定词汇）来做最终的“身份确认”。
        
    - **结果：** 从候选表格中精准提取出唯一的 1 个目标表格；如果分数均不达标，则触发 LLM 慢车道兜底机制。

### 3.3 第三阶段：双模态提取与 LLM 智能兜底 (Dual-Modal Extraction & LLM Fallback)

- **目标**: 针对标准表格执行高效的规则提取；在“定位到章节但未找到标准表格”等复杂、非标场景下，激活大模型（LLM）的泛化与阅读理解能力进行深度信息挖掘。
    
- **功能规格**:
    
    1. **规则快车道 (Rule-Based Fast Track)**: 默认执行链路。对于结构完整、且在 3.2 阶段成功进行语义定位的标准表格，直接基于 Parser 输出的坐标（BBox）和 Pandas 进行行列交叉截取，实现毫秒级的高效提取。
        
    2. **大模型慢车道兜底 (LLM Fallback Extraction)**:
        
        - _触发条件_：当系统在整个文档中**成功定位到了目标表格所在的章节层级**（例如：准确找到了“管理层讨论与分析 -> 主营业务分析”的章节标题），但在该章节范围内**未找到符合预期的目标表格**时触发。这种情况通常由于企业将表格改写成了纯文本段落叙述，或是表格结构极度畸形破裂导致。
            
        - _执行逻辑_：将该章节层级下的所有文本块（Text Blocks）及畸形表格碎片进行拼接，作为上下文（Context）输入给大模型。
            
        - _Prompt 约束_：要求大模型严格按照预定义的 JSON Schema，从这段非结构化的业务描述中“阅读”并抽取出目标财务指标，将其还原为结构化数据输出，如果表格明确未披露，输出`"NOT_DISCLOSED"`。
            
### 3.4 第四阶段：数据归一化与精确语义表达 (Normalization & Semantic Accuracy)

- **目标**: 彻底消除数据歧义，确保最终输出的数据严格符合财务审计“零容错”的底线标准，并完成表级元数据（单位、币种）的智能挂载。
    
- **功能规格**:
    
    1. **“无数据”的精确语义表达 (Null Semantics)**:
        
        - 财务场景下严禁混淆“数值零”与“信息缺失”。系统的输出规范严格定义如下：
            
            - `0.00` (数值零): 表格单元格内或文本中明确填写了“0”。
                
            - `null` (空值): 目标表格已被成功提取，但其内部的某些单元格内容为空白或仅填写了横杠（“-”）。
                
            - `"NOT_DISCLOSED"` (未披露): 找到了表格所在的章节层级，但未找到目标表格，且经过 LLM 兜底分析后，确认上下文中**未包含**该项数据，或者文本明确表示“未披露/不适用”。
                
            - `"NOT_FIND"` (未找到): 整个文档均未找到目标表格应在的章节层级（即目录树中无此模块）。
                
    2. **单位与币种智能提取 (Unit & Currency Extraction & Normalization)**:
        
        - _多锚点寻址_：系统不仅从目标表格内提取数据，还会自动向外围发散，扫描表头（Header）、表格标题（Caption）以及表格上方紧邻的描述文本（如：“单位：元”、“金额单位：人民币万元”）。
            
        - _精准挂载与归一化_：利用正则提取或轻量级实体识别（NER），将识别到的单位信息进行标准化转换（如统一归一化为枚举值 `CNY_YUAN` 或 `CNY_TEN_THOUSAND`）。
            
        - _防范级联错误_：提取出的单位必须作为全局元数据（Metadata）强绑定在当前表格的最终 JSON/DataFrame 结果中，以防止下游数据应用端发生“将万元当成元”的数量级灾难。

#### 3.5 第五阶段：数据后处理与自定义转换策略 (Data Post-processing & Custom Transformation)

- **目标**: 解决“最后一公里”的数据异构问题。针对部分结构极度畸形、存在特殊业务逻辑、或特定公司常年采用反常规排版的表格，提供一层人工干预的脚本化/配置化清洗能力，将非标的中间态数据转换为标准输出格式。
    
- **功能规格**:
    
    1. **可插拔的转换策略引擎 (Pluggable Transformation Engine)**:
        
        - 在系统架构中引入策略注册机制。开发者或业务分析师可以针对特定的 `[表格类型]`编写定制化的 Python 脚本。
            
        - 策略引擎会在第四阶段（归一化）之后自动拦截这些命中规则的数据，将其注入到对应的处理逻辑中。
            
    2. **常见的后处理动作 (Common Transformation Actions)**:
        
        - **行列转置 (Transpose)**: 应对某些企业将标准的“列头”错位写成“行头”的倒置表格。
            
        - **单元格拆分与剥离 (Cell Splitting)**: 应对将多个指标揉捏在一个单元格内的情况（如将“金额”和“同比增速”写在同一个 Cell 里，需通过正则拆分为两列）。
            
        - **多级表头展平 (Multi-index Flattening)**: 将 Parser 解析失败或错位的复杂嵌套表头，通过人工预设的字典映射强制拉平为单层标准键值。
            

### 3.6 第六阶段：全局置信度评分体系 (Confidence Scoring)

每一个提取出的目标表格必须附带一个 `confidence_score` (0-100)。

- **计分规则示例**:
    
    - 基础分：100分。
        
    - 匹配降级：若语义向量匹配度介于 0.85-0.9 之间（非完美匹配），扣 5 分。
        
    - 模型降级：触发 LLM 兜底提取，扣 15 分（因为大模型存在幻觉风险）。
        
    - 单位缺失：未明确找到单位声明（按默认元处理），扣 10 分。
        
- **流转机制**: 得分 `< 85分` 的数据项，自动高亮并推入“人工复核待办队列”。
    

### 3.7 第七阶段：数据溯源与“所见即所得”核对 (Traceability UI/UX)

- **前端展现**: 提取结果页采用“左侧表格名称和结构化表单，右侧 PDF 预览”的双栏布局。
    
- **溯源交互**: 当用户在左侧前端点击目标表格（或点击表格中的某一行）时，右侧 PDF 预览区自动翻页跳转至对应页码，并用高亮红框（Bounding Box）精准圈出整个原始表格（或对应的表格行）的物理位置。
    
- **技术支撑**: 依赖 3.1 阶段保留的 `bbox` 数据传递至前端渲染。
